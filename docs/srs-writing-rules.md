# SRS Writing Rule Book

Reference rules to follow when filling in the SRS template. Keep entries scoped, precise, and audience-appropriate.

## Audience
- Define the audience before writing.
- Determine what the audience needs to learn.
- Fit the documentation to the audience.
- Stay culturally neutral — avoid idioms.

## Document Structure
- State the document's scope and non-scope.
- State the intended audience.
- Summarize key points at the start of each section.
- Compare and contrast where relevant.
- Organize content around the audience's needs.

## Paragraphs
- Keep paragraphs neither too long nor too short.
- Each paragraph should answer: What, Why, and How.
- Focus each paragraph on a single topic.
- Open each paragraph with a strong first sentence.

## Style
- Adopt and follow a style guide; it overrides general grammar rules.
- Write in the second person.
- Place conditions before instructions, not after.
- Format code-related text in code font.
- Use active voice over passive voice; imperative verbs are typically active.
- Choose precise, strong, specific verbs.
- Avoid starting sentences with "There is / There are."
- Minimize adjectives and adverbs.
- Focus each sentence on a single idea.
- Prefer short sentences.
- Cut extraneous words.
- Clarity takes precedence over all other rules.

## Clauses & Grammar
- Reduce subordinate clauses: if a clause extends the same idea, keep it in the sentence; if it branches into a new idea, split it into a new sentence.
- Use "which" for nonessential subordinate clauses (comma before it).
- Use "that" for essential subordinate clauses — the sentence can't stand without it (no comma before it).

## Lists & Tables
- Convert long sentences or dense prose into lists where possible.
- Use bulleted lists for unordered items, numbered lists for ordered items.
- Never use embedded/run-in lists (items stuffed inside a sentence) — bad for technical content.
- Keep list items parallel in structure.
- Start numbered list items with imperative verbs.
- Introduce every list and table with a sentence.
- Build tables only when they're genuinely useful.

## Illustrations
- Write the caption before the illustration.
- Good captions are brief, state the takeaway, and focus the reader's attention.

## Sample Code
- Must build without errors.
- Must perform exactly the task it claims to perform.
- Must follow language-specific conventions.
- Keep it short.

## Prompting an LLM for SRS Content
- Provide source material/context.
- Ask the LLM to impersonate a relevant role.
- State the target audience.
- Specify the document type.
- Define the goal of the section/document.
- Choose the style explicitly.
- Add explicit constraints to the prompt.
